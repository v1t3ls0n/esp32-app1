# 2) 🛠️ הכלים — מי נגד מי

## 2.1 📚 Git — "מכונת זמן" לקוד

**הבעיה:** כשכותבים קוד, כל הזמן משנים אותו. ואז משהו נשבר... ואי אפשר לחזור אחורה. מי לא מכיר את זה מ-Word? קבצים בשם <span dir="ltr"><code>⁨סופי_סופי_באמת2.docx⁩</code></span> 😅

**הפתרון:** Git הוא תוכנה ששומרת **נקודות שמירה** (כמו במשחק מחשב 🎮) של כל תיקיית הקוד.

מושגי היסוד:

- **Repository (ריפו)** 📁 — תיקיית הפרויקט, יחד עם כל ההיסטוריה שלה.
- **Commit (קומיט)** 📸 — "צילום מצב" של הקוד ברגע מסוים, עם הודעה שמסבירה מה השתנה. אפשר תמיד לחזור לכל קומיט.
- **Push (פוש)** ⬆️ — שליחת הקומיטים מהמחשב אל הענן (גיטהאב).

📌 בקיצור: Git = היסטוריה מלאה של הקוד + יכולת לחזור אחורה + עבודה בצוות בלי לדרוס אחד את השני.

## 2.2 ☁️ GitHub — הענן של הקוד

אם Git הוא "מכונת הזמן", אז **GitHub הוא המקום בענן שבו הריפו חי**.

תחשבו על זה כמו Google Drive — אבל לקוד, ועם כל ההיסטוריה:

- הקוד זמין מכל מחשב 💻
- אפשר לשתף אותו עם אחרים 👥
- הוא **מקור האמת** — כל הכלים האחרים (Claude, Vercel) מתחברים אליו

📌 ההבדל בין Git ל-GitHub: **Git** = הטכנולוגיה (מכונת הזמן). **GitHub** = האתר שמארח את הריפו בענן. (כמו ההבדל בין "אימייל" לבין "Gmail".)

## 2.3 🤖 Claude Code — המתכנת שעובד בשבילכם

Claude Code הוא **סוכן AI** (Agent) שמחובר לריפו שלכם ויודע:

- לקרוא ולכתוב קבצי קוד ✍️
- להריץ פקודות ובדיקות 🧪
- לעשות Commit ו-Push לגיטהאב בעצמו ⬆️

ואיך מדברים איתו? **בשפה חופשית.** ככה בדיוק ביקשנו בשיעור:

> "ESP32 שולח הודעה לאפליקציה. האפליקציה שולחת את ההודעה שקיבלה כנוטיפיקיישן למכשיר אנדרואיד שלי"

וקלוד תכנן את הארכיטקטורה, כתב את כל הקבצים, בדק שהכול עובד, ודחף לגיטהאב.

📌 חשוב להבין: קלוד הוא לא קוסם — הוא **עובד אצלכם**. אתם המנהלים: מגדירים מה רוצים, שואלים שאלות, מבקשים שינויים, ובודקים את התוצאה. (בשיעור למשל ביקשנו ממנו באמצע: "בלי ענפים, ענף אחד main וזהו" — והוא סידר את זה.)

## 2.4 🚀 Vercel — מהקוד לאינטרנט, אוטומטית

**הבעיה:** יש לנו קוד בגיטהאב. אבל קוד בגיטהאב זה רק טקסט — איך הוא הופך ל**אתר חי** שהטלפון יכול לגשת אליו?

**הפתרון:** Vercel הוא שירות ענן שמתחבר לריפו בגיטהאב, וכל פעם שנדחף קומיט חדש — הוא:

1. מוריד את הקוד העדכני ⬇️
2. בונה אותו 🏗️
3. מפרסם אותו לכתובת <span dir="ltr"><code>⁨https://...vercel.app⁩</code></span> 🌍

וזה קורה **לבד, תוך בערך דקה, על כל קומיט**. לתהליך הזה קוראים **Deployment** (פריסה).

📌 שימו לב: אף אחד לא "העלה קבצים לשרת" ידנית. פשוט דוחפים קוד לגיטהאב — והאתר מתעדכן. זה נקרא **CI/CD** בעולם המקצועי, וזה הסטנדרט היום.

## 2.5 📡 ESP32 — המוח עם ה-WiFi

את ה-ESP32 אנחנו כבר מכירים מהשיעורים הקודמים — מיקרו-בקר כמו Arduino, אבל עם סופר-כוח: **WiFi מובנה**. זה אומר שהוא יכול לדבר עם שרתים באינטרנט — ובדיוק את זה ניצלנו.

# 5) 🏗️ איך המערכת עובדת מבפנים

עכשיו כשראינו את ה"מה", בואו נבין את ה"איך" — בשפה פשוטה.

## 5.1 📱 מה זה PWA?

**PWA** (Progressive Web App) = אתר אינטרנט שמתנהג כמו אפליקציה:

- מתקינים אותו על מסך הבית (בלי חנות אפליקציות!)
- הוא נפתח במסך מלא, עם אייקון משלו
- והכי חשוב לנו: הוא יכול לקבל **נוטיפיקיישנים**

📌 היתרון הענק: כותבים פעם אחת ב-HTML/JavaScript — ועובד על אנדרואיד, אייפון ומחשב.

## 5.2 👷 מה זה Service Worker?

זה החלק הכי מגניב: **Service Worker** הוא קטע JavaScript שהדפדפן מריץ **ברקע — גם כשהאפליקציה סגורה**.

תחשבו עליו כמו שומר בכניסה לבניין 🏢: גם כשכולם הלכו הביתה, הוא נשאר. וכשמגיעה הודעת Push — הוא מתעורר, ומציג את הנוטיפיקיישן.

בפרויקט שלנו הוא הקובץ <span dir="ltr"><code>⁨public/sw.js⁩</code></span>.

## 5.3 🔔 מה זה Web Push ולמה צריך "מפתחות VAPID"?

הנה קטע מפתיע: השרת שלנו **לא שולח את הנוטיפיקיישן ישירות לטלפון.** למה? כי הטלפון מחליף כתובות רשת כל הזמן (WiFi, סלולר...) ואי אפשר "למצוא" אותו.

במקום זה, לכל דפדפן יש **שירות Push** מרכזי (של גוגל, בכרום) שמחזיק חיבור קבוע לטלפון:

1. כשנרשמנו לנוטיפיקיישנים, הדפדפן נתן לנו **כתובת מנוי** ייחודית
2. השרת שלנו שומר את הכתובת הזו
3. כדי לשלוח — השרת שולח את ההודעה (מוצפנת! 🔒) לשירות של גוגל
4. גוגל דוחפת אותה לטלפון ← ה-Service Worker מתעורר ← 🔔

ומפתחות **VAPID**? זו "תעודת הזהות" של השרת שלנו. בזכותם גוגל יודעת שרק השרת *שלנו* רשאי לשלוח נוטיפיקיישנים למנויים *שלנו* — ואף אחד אחר לא יכול להתחזות.

## 5.4 ⚡ מה זה Serverless ("ללא שרת")?

ב-Vercel אין לנו מחשב שדולק 24/7. במקום זה יש **פונקציות** — כל קובץ בתיקיית <span dir="ltr"><code>⁨api/⁩</code></span> הוא פונקציה שמתעוררת רק כשמגיעה בקשה, רצה שנייה, ונעלמת:

| כתובת | מה היא עושה |
|---|---|
| <span dir="ltr"><code>⁨/api/subscribe⁩</code></span> | רושמת טלפון חדש לרשימת הנוטיפיקיישנים |
| <span dir="ltr"><code>⁨/api/notify⁩</code></span> | מקבלת הודעה מה-ESP32 ושולחת Push לכל הרשומים |
| <span dir="ltr"><code>⁨/api/vapidPublicKey⁩</code></span> | נותנת לאפליקציה את המפתח הציבורי כדי להירשם |
| <span dir="ltr"><code>⁨/api/health⁩</code></span> | "דופק" — בדיקה שהשרת חי וכמה מנויים יש |

✅ יתרונות: חינם (בשימוש ביתי), בלי תחזוקה, נסקייל לבד.

## 5.5 🗄️ למה צריך מסד נתונים (Redis)?

זוכרים שהפונקציות "נעלמות" אחרי כל בקשה? זה אומר שהן **לא זוכרות כלום** בין בקשה לבקשה. אז איפה נשמור את רשימת הטלפונים הרשומים?

בשביל זה חיברנו **Redis** — מסד נתונים קטן ומהיר בענן. הפונקציה <span dir="ltr"><code>⁨subscribe⁩</code></span> כותבת אליו, והפונקציה <span dir="ltr"><code>⁨notify⁩</code></span> קוראת ממנו.

📌 כלל אצבע חשוב בעולם ה-Serverless: **מה שצריך להישמר — נשמר במסד נתונים, לא בזיכרון ולא בקבצים.**

## 5.6 🔑 מה זה DEVICE_TOKEN?

כתובת <span dir="ltr"><code>⁨/api/notify⁩</code></span> פתוחה לאינטרנט — כל אחד יכול לשלוח אליה בקשה. בלי הגנה, כל העולם יוכל להציף אתכם בנוטיפיקיישנים 😱

לכן קבענו **סיסמה משותפת**: ה-ESP32 שולח אותה בכל בקשה, והשרת בודק. בקשה בלי הסיסמה הנכונה נדחית עם שגיאת <span dir="ltr"><code>⁨401 Unauthorized⁩</code></span>.

# 6) 🤖 מה קורה בקוד של ה-ESP32?

לא ניכנס לכל שורה (הקוד המלא בריפו), אבל זו התמצית:

</div>
<div dir="ltr" style="direction:ltr">

```cpp
void setup() {
  connectWiFi();                                    // מתחברים לרשת
  sendNotification("ESP32", "The ESP32 is online! 🚀");  // הודעת "אני חי"
}

void loop() {
  // כשכפתור BOOT נלחץ → שולחים נוטיפיקיישן
  if (buttonPressed()) {
    sendNotification("ESP32", "Button pressed! ⚡");
  }
}
```

</div>
<div dir="rtl" style="direction:rtl">

והפונקציה <span dir="ltr"><code>⁨sendNotification()⁩</code></span> עושה בסך הכול דבר אחד: שולחת בקשת **HTTP POST** לשרת שלנו ב-Vercel, עם ההודעה בפורמט JSON ועם ה-<span dir="ltr"><code>⁨DEVICE_TOKEN⁩</code></span> בכותרת:

</div>
<div dir="ltr" style="direction:ltr">

```json
{ "title": "ESP32", "message": "Button pressed! ⚡" }
```

</div>
<div dir="rtl" style="direction:rtl">

# 7) 📖 מילון מונחים

| מונח | פירוש בשפה פשוטה |
|---|---|
| **Repository (ריפו)** | תיקיית פרויקט + כל היסטוריית השינויים שלה |
| **Commit** | נקודת שמירה של הקוד, עם תיאור מה השתנה |
| **Push** | שליחת הקומיטים מהמחשב לגיטהאב |
| **Deployment** | פרסום גרסה של האפליקציה לאינטרנט |
| **PWA** | אתר שמתנהג כמו אפליקציה ומותקן על מסך הבית |
| **Service Worker** | קוד שרץ ברקע בדפדפן, גם כשהאפליקציה סגורה |
| **Web Push** | מנגנון שליחת נוטיפיקיישנים דרך שירות מרכזי של הדפדפן |
| **VAPID** | זוג מפתחות שמזהה את השרת שלנו מול שירות ה-Push |
| **Serverless** | קוד שרת שרץ כפונקציות לפי דרישה, בלי מחשב דולק 24/7 |
| **Redis** | מסד נתונים קטן ומהיר ששומר מידע בין בקשות |
| **Environment Variables** | ערכים סודיים (מפתחות, סיסמאות) שנשמרים מחוץ לקוד |
| **HTTP POST** | בקשת רשת ששולחת מידע לשרת |
| **JSON** | פורמט טקסט פשוט להעברת מידע מובנה |
| **API** | ה"דלתות" שדרכן תוכנות מדברות אחת עם השנייה |
